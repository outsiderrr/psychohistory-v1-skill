# Engine（引擎）

[English](./README.md) · **中文**

> Psychohistory 的多 Agent 编排与模拟引擎。
> **状态：规划中，尚未实现。** 本目录目前只包含这份 README。

Engine 模块是 Psychohistory 未来的运行时层 —— 在这里，[`../skill/references/schema.md`](../skill/references/schema.md) 定义的场景 schema 将被真正地**执行**，从 agent / relationship 数据产出动态的可能性树。它是 Skill 模块的程序化对应物。

## 为什么要有独立的 Engine

Skill 模块（`../skill/`）定义的是**数据模型和分析方法论** —— 它规范了人类分析师或 AI 助手**应当如何**从一个场景生成一棵可能性树。Skill 今天就能在任何 AI 对话环境里跑起来，只需要 markdown 提示词。

Engine 模块则把同一套流程变成实际的代码 —— 一个运行时，它能：

- 接受一个已校验的 Scenario 对象作为输入
- 程序化地跑完三引擎分析
- 产出结构化的可能性树输出
- 支持交互式事实注入与重算
- 启用那些手工无法实际操作的高级推理模式

## 规划中的能力

### 1. 反向推理模式

给定一个目标结果（例如 *"美联储会在 2026 年 Q4 加息吗？"*），从 agent / relationship 状态**反向追溯** —— 识别哪些前置事件会显著提升该目标的发生概率。

这**不是**正向可能性树的反向运行。它是一个独立的推理子系统，和正向树共享同一套底层数据（agents、relationships、hard constraints），但使用**不同的算法** —— 它搜索的是必要 / 充分条件链，而不是扩展可能性分支。

- **输入**：目标事件 + 当前场景状态
- **输出**：一组排序后的"前置条件路径" —— 每条路径是若干前置事件的链条，如果这些事件发生，会显著改变目标事件的可能性
- **算法草图**：识别目标事件直接依赖的 agent 决策 → 递归追溯每个 agent 的 `concession_triggers` 及其上游驱动 → 一直递归到用户可以监控的可观测事件为止

### 2. 理论验证模式

用户可以提出一个关于"世界如何运作"的理论，引擎对照历史事实和未来预测来检验它。

**关键洞察**：在 Psychohistory 的数据模型里，"一个理论"自然地表达为**对某个或某几个 agent 的认知框架的扰动**。例子：

- *"美联储的真实目标是保护资产价格，不是抑制通胀"* → 把美联储 agent 的 `mental_models` 换成一套替代集
- *"Trump 实际上被 Peter Thiel 通过 Vance 在背后控制"* → 给 Trump 加一个高权重的 `concession_trigger`，锚定在 Vance 的偏好上，并补充 Trump 的心智模型
- *"全球有一个协调中的精英团体"* → 添加一个新的隐藏 agent，定义它对其他 agent 的影响关系

这正是为什么每张角色卡都有一个可选的 `alternative_frameworks` 字段 —— **这就是该模式的结构化入口**。

**验证流程**：

1. 同一场景跑两次 —— 一次用标准 agent 框架，一次用理论扰动后的框架
2. 对比两次运行对已观测到的历史事件的解释力：哪一版更准确地"预测"了真实的过去？
3. 对比两次运行对未来的预测：哪些未来事件能区分两种理论？
4. 输出：关于该理论与历史一致性的判断 + 一组未来事件列表，其发生与否能够区分标准框架和理论扰动框架

### 3. 群体 Agent 升级为一等因果节点

群体 agent（例如 MAGA 基本盘、全球能源市场参与者）在当前 schema 里是作为实体 agent 决策函数**内部**的"变量"出现的。Engine 将把它们升级：当任何一个 `observable_state` 指标（民调平均、市场价格、情绪指数、投票意向）跨越**战略相关阈值**时，这个状态变化会被**提升为可能性树上的一等 Event 节点**。

这让群体驱动的因果（市场恐慌、情绪相变、社会级联）有了明确的结构性位置 —— 而不是被隐藏在单个实体 agent 的决策函数里，难以独立地推理。

结构层的预埋已经在 schema 里做好了：见 [`../skill/references/schema.md`](../skill/references/schema.md) §2.3 中群体 agent 的 `observable_state` 字段。

## 明确不做

Engine 的范围被刻意收窄，以保持诚实。以下方向已经被考虑并在当前阶段放弃：

- **精确百分比概率量化**。这需要一个校准回路 —— 跟踪预测结果并反馈调整 —— 而我们并不具备这个基础设施。基于透明推理的序数排序在当前阶段更诚实
- **预测-结果样本采集基础设施**。上一条的前置条件，同样暂不做
- **外部预测市场数据集成**。诱人，因为我们的 Event 已经是 Polymarket 式的二元命题，格式完美匹配；但这会引入当前阶段不想要的外部依赖

如果项目发展到拥有真实用户群和积累了可观的预测记录，这些方向可以重新评估。**现阶段 Psychohistory 是一个结构化推理工具，而不是概率估计器。**
