# Engine（引擎）

[English](./README.md) · **中文**

> Psychohistory 的多 Agent 编排与模拟引擎。
> **状态：规划中，尚未实现。** 本目录目前只包含这份 README。

Engine 模块是 Psychohistory 未来的运行时层 —— 在这里，[`../skill/references/schema.md`](../skill/references/schema.md) 定义的场景 schema 将被真正地**执行**，从 agent / relationship 数据产出动态的可能性树。它是 Skill 模块的程序化对应物。

## 为什么要有独立的 Engine

Skill 模块（`../skill/`）定义的是**数据模型和分析方法论** —— 它规范了人类分析师或 AI 助手**应当如何**从一个场景生成一棵可能性树。Skill 今天就能在任何 AI 对话环境里跑起来，只需要 markdown 提示词。

Engine 模块则把同一套流程变成实际的代码 —— 一个运行时，它能：

- 接受一个已校验的 Scenario 对象作为输入
- 程序化地跑完三引擎分析
- 产出结构化的可能性树输出
- 支持交互式事实注入与重算
- 启用那些手工无法实际操作的高级推理模式

## 规划中的能力

### 1. 反向推理模式

给定一个目标结果（例如 *"美联储会在 2026 年 Q4 加息吗？"*），从 agent / relationship 状态**反向追溯** —— 识别哪些前置事件会显著提升该目标的发生概率。

这**不是**正向可能性树的反向运行。它是一个独立的推理子系统，和正向树共享同一套底层数据（agents、relationships、hard constraints），但使用**不同的算法** —— 它搜索的是必要 / 充分条件链，而不是扩展可能性分支。

- **输入**：目标事件 + 当前场景状态
- **输出**：一组排序后的"前置条件路径" —— 每条路径是若干前置事件的链条，如果这些事件发生，会显著改变目标事件的可能性
- **算法草图**：识别目标事件直接依赖的 agent 决策 → 递归追溯每个 agent 的 `concession_triggers` 及其上游驱动 → 一直递归到用户可以监控的可观测事件为止

### 2. 理论验证模式（双子模式）

理论验证模式是在**观测到的事件**和**底层 agent 决策模型**之间做映射，通过角色卡上的 `alternative_frameworks` 字段实现（见 `../skill/references/character-schema.md`）。它支持**两个方向**：用户提出假设的验证（**2a**）和系统从观测反推的溯因推理（**2b**）。两个子模式共用同一套验证机制 —— 区别**仅在谁生成候选理论**。

**两个子模式共享的关键洞察**：在 Psychohistory 的数据模型里，"一个理论"自然地表达为**对某个或某几个 agent 的认知框架的扰动**。例子：

- *"美联储的真实目标是保护资产价格，不是抑制通胀"* → 把美联储 agent 的 `mental_models` 换成一套替代集
- *"Trump 实际上被 Peter Thiel 通过 Vance 在背后控制"* → 给 Trump 加一个高权重的 `concession_trigger`，锚定在 Vance 的偏好上，并补充 Trump 的心智模型
- *"全球有一个协调中的精英团体"* → 添加一个新的隐藏 agent，定义它对其他 agent 的影响关系

这正是为什么每张角色卡都有可选的 `alternative_frameworks` 字段 —— **它是两个子模式共同的结构化入口**。

#### 2a. 用户提出的理论验证（假设检验）

用户提出一个具体的理论，引擎检验它。

**输入**：一个替代框架（具体的 `mental_models_override` / `decision_heuristics_override` 等）

**流程**：

1. 同一场景跑两次 —— 一次用标准 agent 框架，一次用理论扰动后的框架
2. 对比两次运行对已观测到的历史事件的解释力：哪一版更准确地"预测"了真实的过去？
3. 对比两次运行对未来的预测：哪些未来事件能区分两种理论？
4. 输出：关于该理论与历史一致性的判断 + 一组未来事件列表，其发生与否能够区分两种世界观

**使用场景**：用户已经有一个具体假设想检验（例如"我觉得美联储的真实目标是 X 不是 Y"）。

#### 2b. 系统生成的溯因推理（观察 → 模型）

不是由用户提供理论，而是**系统从观测到的事件生成候选理论**并检验。这是**溯因推理**（abductive inference）—— 从观察到的结果反推最可能的底层决策模型。

**输入**：同一 agent 的 ≥3-5 个观测事件。

**流程**：

1. 对每个相关 agent，生成 N 个能合理解释这些观测事件的候选替代框架（**多样化假设生成**，不是只挑"最明显"的那个）
2. 通过 Research Hand-off 或已有 references 收集该 agent 的其他已知事实
3. 对每个候选**并行**跑 2a 的验证流程
4. 按**对所有观测事件的总体解释力 + 与硬约束的一致性**给候选排序
5. 输出：排序后的候选列表。最佳候选可能升级为主框架；其他候选作为 `alternative_frameworks` 条目保留

**使用场景**：

- 发生了一个新事件（例如 *"巴基斯坦军方入驻沙特空军基地"*），想理解这个事件揭示了参与方怎样的决策模型
- 多个相关事件缺乏统一解释 → 让系统提出候选理论
- 某个 agent 的官方心智模型和观测到的行为不符 → 系统生成更贴合的替代假设

**认识论风险**（输出的 `honesty_boundaries` 必须显式声明）：

- **欠定性（underdetermination）** —— 单个事件通常有很多合理的模型解释。2b **要求 ≥3-5 个观测事件**做跨事实一致性检验；**单事件溯因推理不可靠**
- **确认偏差陷阱** —— 系统必须显式生成并保留多个候选；**绝不能**一上来就锁定"最明显"的那个
- **模型空间是开放的** —— 真实决策模型可能根本不在生成的候选里。2b 的输出是**"当前数据下的最佳解释"**，而不是因果证明

**和 2a 的关系**：2b 本质是"**在 N 个系统生成的假设上并行跑 2a，然后排序**"。**共用同一套验证后端；区别只在前端的候选生成环节**。它们**不是两个独立子系统**。

### 实施策略

两种子模式共用核心的验证机制。2a 是基础情形（单个用户提供的假设）；2b 在 2a 上面叠加一个候选生成前端。Skill 阶段可以**手动近似 2b**（用户 + AI 一起头脑风暴候选框架 → 一个一个做一致性检验 —— 见长期设计决策原则 14）。Engine 阶段会把候选生成自动化、并行跑验证、按算法排序。

### 3. 群体 Agent 升级为一等因果节点

群体 agent（例如 MAGA 基本盘、全球能源市场参与者）在当前 schema 里是作为实体 agent 决策函数**内部**的"变量"出现的。Engine 将把它们升级：当任何一个 `observable_state` 指标（民调平均、市场价格、情绪指数、投票意向）跨越**战略相关阈值**时，这个状态变化会被**提升为可能性树上的一等 Event 节点**。

这让群体驱动的因果（市场恐慌、情绪相变、社会级联）有了明确的结构性位置 —— 而不是被隐藏在单个实体 agent 的决策函数里，难以独立地推理。

结构层的预埋已经在 schema 里做好了：见 [`../skill/references/schema.md`](../skill/references/schema.md) §2.3 中群体 agent 的 `observable_state` 字段。

### 4. 通过 API 集成自动做研究

把 **Research Hand-off** 的复制粘贴协议（当前 skill 阶段的研究机制，见 `../skill/character-toolkit/SKILL.md` §Step 3.1）替换为**直接的 API 调用**，对接有搜索能力的 LLM。用户一次性配好偏好提供商的 API 密钥，engine 根据任务类型和内容语言自动路由：

- **Perplexity API** —— 英文实证研究（带引用标注）的默认选择
- **OpenAI Search API** —— 通用性提示词
- **Gemini API（Deep Research 模式）** —— 深度多步骤调研
- **Claude API with web search** —— 合成密集型任务
- **Kimi / 豆包 / 元宝 APIs（上线后）** —— 中文内容

**Research Hand-off 作为 fallback 保留**，服务于：

- 不想配 API 密钥的用户（skill 阶段的工作流不受影响继续可用）
- chat UI 特有功能超过 API 能力的场景（某些产品仍然存在这种不对称）
- API 错误、配额耗尽、限流 —— 优雅降级到复制粘贴
- 合规场景：API 使用被限制但 chat UI 被允许

和前面三个能力不同（那些是新的分析模式），这一条是**基础设施升级** —— 它不改变系统能推理什么，只改变获取原材料的效率。skill 阶段的设计和 engine 阶段运行**完全兼容**：同样的 `prompt-0X` Research Hand-off Templates 可以被重新用作 API 调用的 payload，只需要轻量适配。

## 明确不做

Engine 的范围被刻意收窄，以保持诚实。以下方向已经被考虑并在当前阶段放弃：

- **精确百分比概率量化**。这需要一个校准回路 —— 跟踪预测结果并反馈调整 —— 而我们并不具备这个基础设施。基于透明推理的序数排序在当前阶段更诚实
- **预测-结果样本采集基础设施**。上一条的前置条件，同样暂不做
- **外部预测市场数据集成**。诱人，因为我们的 Event 已经是 Polymarket 式的二元命题，格式完美匹配；但这会引入当前阶段不想要的外部依赖

如果项目发展到拥有真实用户群和积累了可观的预测记录，这些方向可以重新评估。**现阶段 Psychohistory 是一个结构化推理工具，而不是概率估计器。**
